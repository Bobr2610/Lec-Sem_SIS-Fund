## 1. Понятие процесса и потока. Модель процесса. Создание процесса, время жизни процесса, завершение процесса

**Процесс** — экземпляр выполняемой программы. У процесса есть собственное виртуальное адресное пространство, набор открытых дескрипторов, права, идентификаторы, контекст выполнения и состояние планирования.

**Поток** — единица выполнения внутри процесса. Потоки одного процесса разделяют адресное пространство и ресурсы процесса, но имеют собственные стек, регистры и состояние выполнения.

### Модель процесса

Типичная модель процесса включает:

- код программы;
- данные и глобальные переменные;
- кучу;
- стек(и) потоков;
- таблицу открытых файлов;
- идентификаторы процесса и родителя;
- права доступа и учетные данные;
- состояние планирования: готов, выполняется, ожидает, остановлен, завершен/zombie.

В терминах ядра Linux процесс представлен дескриптором процесса `struct task_struct`. Эта структура содержит почти всю информацию, которая нужна ядру для управления задачей:

- состояние процесса;
- идентификаторы PID и связи с родителем/потомками;
- приоритет и параметры планирования;
- указатели на адресное пространство;
- сведения об открытых файлах;
- обработчики и ожидающие сигналы;
- учетную информацию и права;
- ссылки, связывающие процесс со списками задач ядра.

В Linux поток не является принципиально другой сущностью ядра: и процессы, и потоки представлены задачами. Различие определяется тем, какие ресурсы разделяются между задачами. Например, потоки одного процесса разделяют адресное пространство и многие ресурсы, но имеют разные стеки и контекст выполнения.

### Создание процесса

На Unix-подобных системах новый процесс обычно создается через `fork()`.

- `fork()` создает копию процесса-родителя.
- После `fork()` у родителя и потомка разные PID.
- Современные Unix-системы обычно используют copy-on-write: страницы памяти реально копируются только при записи.
- Часто сразу после `fork()` потомок вызывает `execve()`, чтобы заменить образ процесса другой программой.

В Linux семейство вызовов создания задач опирается на механизм `clone()`: он позволяет точно указать, какие ресурсы новая задача будет разделять с родителем. На этом строится различие между классическим процессом после `fork()` и потоком, который разделяет адресное пространство, файловые дескрипторы и другие ресурсы.

### Время жизни процесса

Процесс существует от момента создания до завершения.

- Пока процесс работает, он может создавать дочерние процессы.
- Если родитель завершился раньше потомка, потомок становится сиротой и обычно переходит под управление `init`/`systemd`.

### Завершение процесса

Процесс завершает работу:

- явно через `exit()` / `return` из `main()`;
- по сигналу;
- из-за аварийной ошибки.

После завершения процесс может временно остаться в состоянии zombie: код уже не выполняется, но запись в таблице процессов хранит статус завершения. Родитель должен вызвать `wait()` / `waitpid()` и забрать этот статус.

С точки зрения ядра завершение означает освобождение ресурсов процесса и перевод задачи в состояние, из которого родитель может получить код завершения. Если родитель не забирает статус, остается zombie-запись: она уже не занимает пользовательское адресное пространство, но нужна для передачи результата завершения.

## 2. Межпроцессное взаимодействие. Состязательные ситуации. Критические области

**Межпроцессное взаимодействие (IPC)** — обмен данными и синхронизация между процессами.

Основные механизмы IPC:

- очереди сообщений System V;
- семафоры System V;
- разделяемая память System V;
- каналы, FIFO, сокеты, сигналы, файлы и `mmap()`.

### Состязательные ситуации

Состязательная ситуация возникает, когда несколько процессов или потоков одновременно обращаются к общему ресурсу, и результат зависит от порядка выполнения.

Это приводит к ошибкам:

- потеря обновлений;
- чтение несогласованных данных;
- нарушение инвариантов структуры данных.

### Критическая область

Критическая область — участок кода, который обращается к общему ресурсу и должен выполняться взаимно исключающим образом.

Требования к решению:

- взаимное исключение;
- отсутствие тупиков;
- отсутствие голодания;
- ограниченное ожидание.

Критическую область обычно защищают семафорами, мьютексами, спин-блокировками или другими примитивами синхронизации.

Синхронизация ядра нужна для защиты разделяемых данных ядра от одновременного доступа. Для коротких критических секций в ядре часто используются спин-блокировки, особенно когда нельзя спать. Если поток выполнения может блокироваться и ждать, применяются семафоры или мьютексы. Для пользовательских процессов в этих вопросах основное внимание уделяется System V IPC, но общая проблема та же: нужно защитить общий ресурс от гонок.

## 3. Взаимодействие процессов с помощью очередей сообщений. Очереди сообщений System V

Очередь сообщений System V — механизм IPC, где сообщения хранятся в ядре, а процессы отправляют и принимают их через идентификатор очереди.

### Назначение

- асинхронная передача данных;
- буферизация сообщений в ядре;
- разделение отправителя и получателя по времени.

### Создание и получение доступа

```c
int msgget(key_t key, int msgflg);
```

- `key` — ключ очереди, то есть имя IPC-объекта в пространстве System V;
- `msgflg` — флаги доступа и создания.

Основные флаги:

- `IPC_CREAT` — создать очередь, если ее нет;
- `IPC_EXCL` — создать только новую очередь, иначе ошибка;
- права доступа в младших 9 битах.

### Отправка сообщения

```c
int msgsnd(int msqid, const void *msgp, size_t msgsz, int msgflg);
```

Сообщение обычно имеет вид:

```c
struct msgbuf {
	long mtype;
	char mtext[...];
};
```

- `mtype` — положительный тип сообщения;
- `mtext` — данные сообщения;
- `msgsz` в `msgsnd()` и `msgrcv()` задает размер поля `mtext`, а не всей структуры вместе с `mtype`.

### Прием сообщения

```c
ssize_t msgrcv(int msqid, void *msgp, size_t msgsz, long msgtyp, int msgflg);
```

Смысл `msgtyp`:

- `0` — первое сообщение в очереди;
- `> 0` — первое сообщение указанного типа;
- `< 0` — сообщение с наименьшим типом, не превосходящим `|msgtyp|`; если таких несколько, берется самое старое среди сообщений этого типа.

Полезные флаги:

- `IPC_NOWAIT` — не блокироваться, если операция сейчас невозможна;
- `MSG_NOERROR` — при приеме обрезать слишком длинное сообщение вместо ошибки.

### Управление очередью

```c
int msgctl(int msqid, int cmd, struct msqid_ds *buf);
```

Команды:

- `IPC_STAT` — получить атрибуты очереди;
- `IPC_SET` — изменить атрибуты;
- `IPC_RMID` — удалить очередь.

### Свойства

- очередь хранится в ядре;
- процессы не обязаны одновременно быть активны;
- сообщения имеют тип и данные;
- можно реализовать приоритет по типам;
- очередь существует, пока ее явно не удалят через `msgctl(..., IPC_RMID, ...)` или пока система не будет перезагружена.

### Примеры из мануалов

Официальные справочные страницы подтверждают: `msgget()` создает или открывает очередь, `msgsnd()` отправляет, `msgrcv()` получает, а `msgctl()` управляет очередью.

## 4. Синхронизация процессов с помощью семафоров System V

**Семафор** — счетчик, используемый для синхронизации доступа к ресурсу.

### Назначение

- взаимное исключение;
- ожидание события;
- координация процессов.

### Основные свойства

- значение семафора — целое число;
- операции над семафором атомарны;
- `semop()` может блокировать процесс, пока условие не станет выполнимым.

### Создание и получение доступа

```c
int semget(key_t key, int nsems, int semflg);
```

Флаги:

- `IPC_CREAT` — создать набор, если его нет;
- `IPC_EXCL` — ошибка, если набор уже существует;
- права доступа в младших 9 битах.

### Операции над семафорами

```c
int semop(int semid, struct sembuf *sops, size_t nsops);
```

Структура операции:

```c
struct sembuf {
	unsigned short sem_num;
	short sem_op;
	short sem_flg;
};
```

- `sem_op < 0` — уменьшить семафор на `|sem_op|`; если текущее значение меньше нужного и нет `IPC_NOWAIT`, процесс ждет;
- `sem_op = 0` — ждать, пока значение станет нулем;
- `sem_op > 0` — увеличить значение семафора.

Флаги:

- `IPC_NOWAIT` — не блокироваться, если операция не может быть выполнена сразу;
- `SEM_UNDO` — ядро запоминает изменение и автоматически компенсирует его при завершении процесса.

### Управление набором семафоров

```c
int semctl(int semid, int semnum, int cmd, ...);
```

Основные команды:

- `IPC_STAT` — получить информацию;
- `IPC_SET` — изменить управляющие параметры;
- `IPC_RMID` — удалить набор;
- `GETVAL` — получить значение семафора;
- `SETVAL` — установить значение;
- `GETALL` / `SETALL` — получить/установить все значения;
- `GETNCNT` — число процессов, ожидающих увеличения значения;
- `GETZCNT` — число процессов, ожидающих нулевого значения.

### Поведение

- набор семафоров создается в ядре;
- значение семафора задает состояние синхронизации;
- для инициализации часто используют `semctl(..., SETVAL/SETALL, ...)`.

### Пример из лекции

Для сервер-клиентной синхронизации в лекции используется набор из двух семафоров. Начальное состояние `0 | 1`: первый семафор сигнализирует о наличии запроса для сервера, второй ограничивает доступ клиентов к разделяемому ресурсу.

## 5. Взаимодействие процессов с помощью разделяемой памяти System V

**Разделяемая память** — область памяти, доступная нескольким процессам одновременно.

Это самый быстрый механизм IPC для обмена большими объемами данных, потому что копирование через ядро минимально.

### Создание

```c
int shmget(key_t key, size_t size, int shmflg);
```

- `IPC_CREAT` — создать сегмент;
- `IPC_EXCL` — ошибка, если сегмент уже существует;
- права доступа задаются в младших 9 битах `shmflg`.

### Присоединение

```c
void *shmat(int shmid, const void *shmaddr, int shmflg);
```

- возвращает адрес, по которому сегмент отображен в адресное пространство процесса;
- при ошибке возвращает `(void *) -1`;
- флаг `SHM_RDONLY` присоединяет сегмент только для чтения.

### Отсоединение

```c
int shmdt(const void *shmaddr);
```

- отсоединяет сегмент от процесса.

### Управление

```c
int shmctl(int shmid, int cmd, struct shmid_ds *buf);
```

Команды:

- `IPC_STAT` — получить атрибуты;
- `IPC_SET` — изменить атрибуты;
- `IPC_RMID` — пометить сегмент на удаление.

### Свойства

- несколько процессов видят один и тот же участок данных;
- для согласованного доступа нужна синхронизация, чаще всего семафорами;
- `shmdt()` только отсоединяет сегмент от конкретного процесса;
- `IPC_RMID` помечает сегмент на удаление, а фактически память освобождается после последнего отсоединения.

## 6. Алгоритм синхронизации процессов на семафорах и разделяемой памяти

Типовая схема из лекции: один серверный процесс и несколько клиентских. Используются разделяемый ресурс и набор из двух семафоров с начальными значениями `0 | 1`.

Идея:

- разделяемая память используется как буфер данных;
- семафоры используются как сигналы состояния.

### Схема работы

1. Сервер создает разделяемую память и набор из двух семафоров, затем устанавливает значения `0 | 1`.
2. Сервер ждет данных от клиента операцией над первым семафором: условно `semop(-1, 0)`.
3. Клиент захватывает доступ к разделяемой памяти через второй семафор: `semop(0, -1)`, состояние становится `0 | 0`.
4. Клиент записывает запрос и сигнализирует серверу: `semop(1, 0)`, состояние становится `1 | 0`.
5. Клиент ждет ответа сервера операцией `semop(-1, -1)`.
6. Сервер читает запрос, обрабатывает его, записывает ответ и выполняет `semop(1, 1)`, состояние становится `1 | 1`.
7. Сервер снова переходит к ожиданию нового запроса.
8. Клиент разблокируется, читает ответ и освобождает доступ к ресурсу: `semop(0, 1)`, состояние возвращается к `0 | 1`.

### Зачем нужны оба механизма

- разделяемая память дает быстрый обмен данными;
- семафоры обеспечивают правильный порядок доступа.

Без семафоров возможны гонки: один процесс прочитает данные до того, как другой их полностью запишет.

## 7. Файлы и файловая система. Виртуальная файловая система (VFS): определение и назначение. Описание объектов VFS

**Файл** — именованная область данных и метаданных в файловой системе.

**Файловая система** — способ хранения, организации и доступа к файлам.

**VFS** (Virtual File System) — абстрактный слой ядра, который предоставляет единый интерфейс для разных реальных файловых систем.

VFS можно рассматривать как объектную модель, реализованную на C: ядро работает с набором общих объектов и таблиц операций, а конкретная файловая система подставляет свои функции. Благодаря этому системные вызовы вроде `open()`, `read()` и `write()` не зависят напрямую от ext4, tmpfs, procfs или другой конкретной файловой системы.

### Назначение VFS

- скрыть различия между ext4, xfs, tmpfs и другими ФС;
- дать единый интерфейс для `open`, `read`, `write`, `close` и т.д.;
- связать операции с объектами файловой системы.

### Основные объекты VFS

- **Суперблок** — информация о смонтированной файловой системе в целом;
- **inode** (файловый индекс, индексный дескриптор) — метаданные объекта файловой системы без имени;
- **dentry** — элемент каталога, связывающий имя с inode и ускоряющий разрешение путей;
- **file** — объект открытого файла: режим открытия, текущая позиция, ссылки на операции и inode/dentry.

## 8. Операции над объектами VFS

Каждый объект VFS имеет набор операций.

### Для суперблока

- создание и уничтожение inode;
- запись информации о ФС;
- синхронизация состояния;

- `rename()` — переименовать;
- `chmod()`, `chown()` — изменить права и владельца.

### Что происходит при `open()` и `close()`

#### Последовательность шагов функции `open()`

1. Ядро выполняет поиск в таблице дескрипторов файлов процесса первой незадействованной позиции. Номер (индекс) этой позиции является возвращаемым значением функции `open`.
2. Ядро выполняет поиск незанятой позиции в таблице открытых файлов ядра.
3. Если свободная позиция найдена, в найденную запись таблицы открытых файлов процесса заносится ссылка на найденную запись таблицы ядра.
4. В таблицу открытых файлов в найденную позицию записывается ссылка на ту запись из таблицы inode, которая соответствует нужному файлу.
5. В таблице открытых файлов индексируется указатель текущей позиции чтения/записи.
6. В запись таблицы открытых файлов записывается режим открытия файла.
7. В записи таблицы открытых файлов изменяется счетчик ссылок на данный файл.
8. Запись счетчика ссылок индексного дескриптора увеличивается на 1.

#### Последовательность шагов функции `close()`

1. Соответствующая запись в таблице открытых файлов процесса помечается как неиспользуемая (незадействованная).
2. Ядром уменьшается значение счетчика ссылок таблицы открытых файлов на 1.
   - Если значение ссылок отлично от 0, перейти к пункту 6.
3. Позиция таблицы открытых файлов помечается как незадействованная.
4. Значение счетчика ссылок в соответствующей записи таблицы индексных дескрипторов файла уменьшается на 1.
   - Если значение счетчика ссылок отлично от 0, перейти к пункту 6.
5. Если значение счетчика ссылок в записи таблицы индексных дескрипторов отлично от 0, то происходит возвращение управления вызвавшему процессу с кодом 0. Иначе ядро помечает соответствующую запись в таблице индексных дескрипторов как неактивную и физически удаляет память, которая выделена на диске.
6. Ядро возвращает управление вызвавшему процессу и код успешного выполнения 0.

#### Примечание про `unlink()`

`unlink()` удаляет имя из каталога, но данные файла физически удаляются только когда нет ни жестких ссылок, ни открытых файловых описаний (счетчик ссылок в inode стал равен 0).

## 10. Файлы, проецируемые в память. Обмен данными между процессами с помощью файлов, проецируемых в память

**Отображение файла в память** — это создание соответствия между участком виртуальной памяти и содержимым файла.

### Основные вызовы

```c
void *mmap(void *addr, size_t length, int prot, int flags, int fd, off_t offset);
int munmap(void *addr, size_t length);
```

### Зачем это нужно

- быстрый доступ к файлам через обычные операции с памятью;
- удобный обмен данными между процессами через один и тот же отображенный файл;
- меньше явных вызовов `read`/`write`.

### Типы отображения

- `MAP_SHARED` — изменения видны другим процессам и могут попасть в файл;
- `MAP_PRIVATE` — copy-on-write, изменения не идут в исходный файл.

### Обмен данными между процессами

Если два процесса отображают один и тот же файл с `MAP_SHARED`, то они могут обмениваться данными через общую область памяти.

Для согласованности обычно дополнительно используют:

- семафоры;
- барьеры;
- мьютексы, размещенные в разделяемой памяти;
- `msync()` при необходимости принудительно синхронизировать изменения с файлом на диске.

Важно различать видимость изменений между процессами и запись на диск: `MAP_SHARED` делает изменения общими для отображений, а `msync()` нужен именно для управления сбросом измененных страниц в файл.

### Источник

Справка `mmap(2)` подтверждает, что отображение создается в виртуальном адресном пространстве процесса, а `munmap()` удаляет отображение.

## 11. Алгоритмы управления динамической памятью

### First fit
Первый свободный блок размером ≥ запроса. Просмотр списка с начала, при первом подходящем — занимаем (с разбиением, если блок сильно больше).

```cpp
struct B { size_t sz; bool fr; B* n; };
B* free = nullptr;

B* first_fit(size_t n) {
    for (B* p = free; p; p = p->n)
        if (p->sz >= n) { p->fr = 0; return p; }
    return nullptr;
}
```

### Best fit
Свободный блок с минимальным `sz ≥ n`. Требует полного просмотра списка, оставляет мелкие остатки.

```cpp
B* best_fit(size_t n) {
    B* b = nullptr;
    for (B* p = free; p; p = p->n)
        if (p->sz >= n && (!b || p->sz < b->sz)) b = p;
    if (b) b->fr = 0;
    return b;
}
```

### Worst fit
Свободный блок с максимальным `sz`. Остаток после разбиения, вероятно, тоже пригоден.

```cpp
B* worst_fit(size_t n) {
    B* b = nullptr;
    for (B* p = free; p; p = p->n)
        if (p->sz >= n && (!b || p->sz > b->sz)) b = p;
    if (b) b->fr = 0;
    return b;
}
```

### Освобождение в рассортированном списке
Свободные блоки хранятся по возрастанию адреса. При освобождении вставляем на место и сливаем с физическими соседями слева/справа.

```cpp
struct N { size_t sz; N *p, *n; char* a; };
N* h = nullptr;

void free_sorted(N* b) {
    N* prev = nullptr, *cur = h;
    while (cur && cur->a < b->a) { prev = cur; cur = cur->n; }
    b->n = cur; b->p = prev;
    if (prev) prev->n = b; else h = b;
    if (cur) cur->p = b;
    if (prev && prev->a + prev->sz == b->a) {
        prev->sz += b->sz; prev->n = b->n; if (b->n) b->n->p = prev; b = prev;
    }
    if (b->n && b->a + b->sz == b->n->a) {
        b->sz += b->n->sz; b->n = b->n->n; if (b->n) b->n->p = b;
    }
}
```

### Boundary tags
В начале и конце блока хранят `size` и `free`. По «подвалу» слева за O(1) находим начало предыдущего блока и сливаемся с соседями.

```cpp
struct T { size_t sz; bool fr;
    size_t* foot() { return (size_t*)((char*)this + sz - sizeof(size_t)); }
};
T* prev(T* b) { size_t* f = (size_t*)((char*)b - sizeof(size_t)); return (T*)((char*)b - *f); }
T* next(T* b) { return (T*)((char*)b + b->sz); }

void free_bt(T* b) {
    b->fr = 1; *b->foot() = b->sz;
    T* p = prev(b); if (p && p->fr) { p->sz += b->sz; *p->foot() = p->sz; b = p; }
    T* n = next(b); if (n && n->fr) { b->sz += n->sz; *b->foot() = b->sz; }
}
```

### Buddy system
Память — блоки размером `2^k`. Нехватка блока порождает деление большего пополам. При освобождении двойник находится XOR'ом адреса с размером; если свободен — сливаем.

```cpp
struct Buddy {
    struct B { size_t sz; B* n; };
    std::vector<B*> f; // f[k] — блоки размера 2^k

    void* alloc(size_t r) {
        int k = 0; while ((1u << k) < r) ++k;
        int j = k; while (j < (int)f.size() && !f[j]) ++j;
        if (j == (int)f.size()) return nullptr;
        B* b = f[j]; f[j] = b->n;
        while (j > k) { --j; B* d = (B*)((char*)b + (1u << j)); d->sz = 1u << j; d->n = f[j]; f[j] = d; }
        return b;
    }
    void free(void* p, size_t r) {
        int k = 0; while ((1u << k) < r) ++k;
        B* b = (B*)p; b->sz = 1u << k;
        while (k < (int)f.size()) {
            B* d = (B*)((size_t)b ^ (1u << k));
            B** q = &f[k]; while (*q && *q != d) q = &(*q)->n;
            if (!*q) { b->n = f[k]; f[k] = b; return; }
            *q = (*q)->n; if (d < b) b = d; ++k;
        }
    }
};
```

### Двусвязный список + RB-дерево
Все блоки связаны в двусвязный список по адресу (для слияния соседей). Свободные блоки индексируются красно-чёрным деревом по размеру — поиск best fit за `O(log m)`. `std::multimap` в C++ реализован на RB-дереве.

```cpp
struct Block { size_t sz; bool fr; Block *pr, *nx; std::multimap<size_t, Block*>::iterator it; };
std::multimap<size_t, Block*> t;

Block* alloc(size_t n) {
    auto it = t.lower_bound(n);
    if (it == t.end()) return nullptr;
    Block* b = it->second; t.erase(it); b->fr = 0;
    if (b->sz > n + sizeof(Block)) {
        Block* r = (Block*)((char*)b + n);
        r->sz = b->sz - n; r->fr = 1; r->pr = b; r->nx = b->nx;
        if (b->nx) b->nx->pr = r; b->nx = r; b->sz = n;
        r->it = t.insert({r->sz, r});
    }
    return b;
}

void free_rb(Block* b) {
    b->fr = 1;
    if (b->pr && b->pr->fr) {
        t.erase(b->pr->it); b->pr->sz += b->sz;
        b->pr->nx = b->nx; if (b->nx) b->nx->pr = b->pr; b = b->pr;
    }
    if (b->nx && b->nx->fr) {
        t.erase(b->nx->it); b->sz += b->nx->sz;
        b->nx = b->nx->nx; if (b->nx) b->nx->pr = b;
    }
    b->it = t.insert({b->sz, b});
}
```

## 12. Сбалансированные по высоте деревья поиска. Повороты. B-дерево, B+-дерево, B*-дерево, B*+-дерево

**АВЛ-дерево** — ДБП, у которого для любой вершины |h(left) − h(right)| ≤ 1. Баланс-фактор = h(left) − h(right) ∈ {−1, 0, 1}. Балансировка — 4 поворота.

```cpp
template <typename Key>
struct AVLNode {
    Key key;
    int height;
    AVLNode *left, *right;
};

int h(AVLNode* n) { return n ? n->height : 0; }
void upd(AVLNode* n) { if (n) n->height = 1 + std::max(h(n->left), h(n->right)); }

AVLNode* smallRight(AVLNode* y) {   // LL
    AVLNode* x = y->left;
    y->left = x->right;
    x->right = y;
    upd(y); upd(x);
    return x;
}
AVLNode* smallLeft(AVLNode* x) {    // RR
    AVLNode* y = x->right;
    x->right = y->left;
    y->left = x;
    upd(x); upd(y);
    return y;
}
AVLNode* bigRight(AVLNode* n) {     // LR
    n->left = smallLeft(n->left);
    return smallRight(n);
}
AVLNode* bigLeft(AVLNode* n) {      // RL
    n->right = smallRight(n->right);
    return smallLeft(n);
}
```

**B-дерево** — сильно ветвящееся дерево, все листья на одной глубине. Узел (кроме корня) содержит от t−1 до 2t−1 ключей, у внутреннего узла k ключей — k+1 детей.  
**B+** — данные только в листьях, листья связаны в список.  
**B\*** — минимальное заполнение 2/3, при переполнении сначала redistribute с соседом, иначе split 2→3.  
**B\*+** — объединяет B+ и B\*.

### Узел и поиск (общий для всех)

```cpp
template <typename Key>
struct BNode {
    bool is_leaf;
    std::vector<Key> keys;
    std::vector<BNode*> children;
    BNode* next_leaf;   // B+, B*+
};

BNode<Key>* search(BNode<Key>* n, Key k) {
    if (!n) return nullptr;
    int i = 0;
    while (i < n->keys.size() && k > n->keys[i]) ++i;
    if (i < n->keys.size() && n->keys[i] == k) return n;
    if (n->is_leaf) return nullptr;
    return search(n->children[i], k);
}
```

### Вставка: сплит, redistribute, split2to3

```cpp
// Обычный сплит 1->2 (для B, B+, B*+ при росте корня)
void splitChild(BNode<Key>* p, int i) {
    BNode<Key>* old = p->children[i];
    BNode<Key>* neo = new BNode<Key>{old->is_leaf};
    int mid = old->keys.size() / 2;
    int up = old->keys[mid];

    if (old->is_leaf) {
        neo->keys.assign(old->keys.begin() + mid, old->keys.end());
        old->keys.erase(old->keys.begin() + mid, old->keys.end());
        neo->next_leaf = old->next_leaf;
        old->next_leaf = neo;
    } else {
        neo->keys.assign(old->keys.begin() + mid + 1, old->keys.end());
        old->keys.erase(old->keys.begin() + mid, old->keys.end());
        neo->children.assign(old->children.begin() + mid + 1, old->children.end());
        old->children.erase(old->children.begin() + mid + 1, old->children.end());
    }
    p->keys.insert(p->keys.begin() + i, up);
    p->children.insert(p->children.begin() + i + 1, neo);
}

// Перераспределение через родителя (B*, B*+)
bool redistribute(BNode<Key>* p, int i, int max_keys) {
    BNode<Key>* node = p->children[i];
    if (i + 1 < p->children.size()) {
        BNode<Key>* r = p->children[i + 1];
        if (r->keys.size() < max_keys) {
            r->keys.insert(r->keys.begin(), p->keys[i]);
            p->keys[i] = node->keys.back();
            node->keys.pop_back();
            if (!node->is_leaf) {
                r->children.insert(r->children.begin(), node->children.back());
                node->children.pop_back();
            }
            return true;
        }
    }
    if (i > 0) {
        BNode<Key>* l = p->children[i - 1];
        if (l->keys.size() < max_keys) {
            l->keys.push_back(p->keys[i - 1]);
            p->keys[i - 1] = node->keys[0];
            node->keys.erase(node->keys.begin());
            if (!node->is_leaf) {
                l->children.push_back(node->children[0]);
                node->children.erase(node->children.begin());
            }
            return true;
        }
    }
    return false;
}

// B* сплит 2->3: два полных узла + разделитель -> три узла
void split2to3(BNode<Key>* p, int i) {
    BNode<Key>* L = p->children[i];
    BNode<Key>* R = p->children[i + 1];
    BNode<Key>* M = new BNode<Key>{L->is_leaf};

    std::vector<Key> all;
    all.insert(all.end(), L->keys.begin(), L->keys.end());
    all.push_back(p->keys[i]);
    all.insert(all.end(), R->keys.begin(), R->keys.end());

    int n = all.size(), s1 = n / 3, s2 = 2 * n / 3;
    L->keys.assign(all.begin(), all.begin() + s1);
    M->keys.assign(all.begin() + s1 + 1, all.begin() + s2);
    R->keys.assign(all.begin() + s2 + 1, all.end());
    p->keys[i] = all[s1];

    if (!L->is_leaf) {
        std::vector<BNode<Key>*> allc;
        allc.insert(allc.end(), L->children.begin(), L->children.end());
        allc.insert(allc.end(), R->children.begin(), R->children.end());
        int c1 = s1 + 1, c2 = s2 + 1;
        L->children.assign(allc.begin(), allc.begin() + c1);
        M->children.assign(allc.begin() + c1, allc.begin() + c2);
        R->children.assign(allc.begin() + c2, allc.end());
    } else {
        M->next_leaf = R->next_leaf;
        L->next_leaf = M;
    }
    p->keys.insert(p->keys.begin() + i + 1, all[s2]);
    p->children.insert(p->children.begin() + i + 1, M);
}
```

### Удаление: borrow + merge

```cpp
void borrowFromPrev(BNode<Key>* p, int i) {
    BNode<Key>* node = p->children[i];
    BNode<Key>* left = p->children[i - 1];
    node->keys.insert(node->keys.begin(), p->keys[i - 1]);
    p->keys[i - 1] = left->keys.back();
    left->keys.pop_back();
}

void mergeNodes(BNode<Key>* p, int i) {
    BNode<Key>* L = p->children[i];
    BNode<Key>* R = p->children[i + 1];
    L->keys.push_back(p->keys[i]);
    for (auto k : R->keys) L->keys.push_back(k);
    if (L->is_leaf) L->next_leaf = R->next_leaf;
    p->keys.erase(p->keys.begin() + i);
    p->children.erase(p->children.begin() + i + 1);
    delete R;
}
```

### B*+ дерево — полный класс с insert

```cpp
class BStarPlusTree {
    static const int T = 2;                 // max_keys = 2T
    BNode<int>* root;

    bool full(BNode<int>* n) { return n->keys.size() >= 2 * T; }

    void splitChild(BNode<int>* p, int i) {
        BNode<int>* old = p->children[i];
        BNode<int>* neo = new BNode<int>{old->is_leaf};
        int mid = old->keys.size() / 2;
        int up = old->keys[mid];

        if (old->is_leaf) {
            neo->keys.assign(old->keys.begin() + mid, old->keys.end());
            old->keys.erase(old->keys.begin() + mid, old->keys.end());
            neo->next_leaf = old->next_leaf;
            old->next_leaf = neo;
        } else {
            neo->keys.assign(old->keys.begin() + mid + 1, old->keys.end());
            old->keys.erase(old->keys.begin() + mid, old->keys.end());
            neo->children.assign(old->children.begin() + mid + 1, old->children.end());
            old->children.erase(old->children.begin() + mid + 1, old->children.end());
        }
        p->keys.insert(p->keys.begin() + i, up);
        p->children.insert(p->children.begin() + i + 1, neo);
    }

    bool tryRedistribute(BNode<int>* p, int i) {
        BNode<int>* node = p->children[i];
        if (i + 1 < p->children.size()) {
            BNode<int>* r = p->children[i + 1];
            if (!full(r)) {
                r->keys.insert(r->keys.begin(), p->keys[i]);
                if (!node->is_leaf) {
                    r->children.insert(r->children.begin(), node->children.back());
                    node->children.pop_back();
                }
                p->keys[i] = node->keys.back();
                node->keys.pop_back();
                return true;
            }
        }
        if (i > 0) {
            BNode<int>* l = p->children[i - 1];
            if (!full(l)) {
                l->keys.push_back(p->keys[i - 1]);
                if (!node->is_leaf) {
                    l->children.push_back(node->children[0]);
                    node->children.erase(node->children.begin());
                }
                p->keys[i - 1] = node->keys[0];
                node->keys.erase(node->keys.begin());
                return true;
            }
        }
        return false;
    }

    void split2to3(BNode<int>* p, int i) {
        BNode<int>* L = p->children[i];
        BNode<int>* R = p->children[i + 1];
        BNode<int>* M = new BNode<int>{L->is_leaf};

        std::vector<int> all;
        all.insert(all.end(), L->keys.begin(), L->keys.end());
        all.push_back(p->keys[i]);
        all.insert(all.end(), R->keys.begin(), R->keys.end());

        int n = all.size(), s1 = n / 3, s2 = 2 * n / 3;
        L->keys.assign(all.begin(), all.begin() + s1);
        M->keys.assign(all.begin() + s1 + 1, all.begin() + s2);
        R->keys.assign(all.begin() + s2 + 1, all.end());
        p->keys[i] = all[s1];

        if (!L->is_leaf) {
            std::vector<BNode<int>*> allc;
            allc.insert(allc.end(), L->children.begin(), L->children.end());
            allc.insert(allc.end(), R->children.begin(), R->children.end());
            int c1 = s1 + 1, c2 = s2 + 1;
            L->children.assign(allc.begin(), allc.begin() + c1);
            M->children.assign(allc.begin() + c1, allc.begin() + c2);
            R->children.assign(allc.begin() + c2, allc.end());
        } else {
            M->next_leaf = R->next_leaf;
            L->next_leaf = M;
        }
        p->keys.insert(p->keys.begin() + i + 1, all[s2]);
        p->children.insert(p->children.begin() + i + 1, M);
    }

    void insertNonFull(BNode<int>* n, int key) {
        if (n->is_leaf) {
            int i = 0;
            while (i < n->keys.size() && key > n->keys[i]) ++i;
            n->keys.insert(n->keys.begin() + i, key);
        } else {
            int i = (int)n->keys.size() - 1;
            while (i >= 0 && key < n->keys[i]) --i;
            ++i;
            if (full(n->children[i])) {
                if (!tryRedistribute(n, i)) split2to3(n, i);
                if (key > n->keys[i]) ++i;
            }
            insertNonFull(n->children[i], key);
        }
    }

public:
    BStarPlusTree() : root(new BNode<int>{true}) {}

    void insert(int key) {
        if (full(root)) {
            BNode<int>* neo = new BNode<int>{false};
            neo->children.push_back(root);
            splitChild(neo, 0);
            root = neo;
        }
        insertNonFull(root, key);
    }

    bool search(int key) {
        BNode<int>* n = root;
        while (n) {
            int i = 0;
            while (i < n->keys.size() && key > n->keys[i]) ++i;
            if (i < n->keys.size() && n->keys[i] == key) return true;
            if (n->is_leaf) return false;
            n = n->children[i];
        }
        return false;
    }
};
```
