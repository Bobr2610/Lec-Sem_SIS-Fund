## 1. Понятие процесса и потока. Модель процесса. Создание процесса, время жизни процесса, завершение процесса

**Процесс** — экземпляр выполняемой программы. У процесса есть собственное виртуальное адресное пространство, набор открытых дескрипторов, права, идентификаторы, контекст выполнения и состояние планирования.

**Поток** — единица выполнения внутри процесса. Потоки одного процесса разделяют адресное пространство и ресурсы процесса, но имеют собственные стек, регистры и состояние выполнения.

### Модель процесса

Типичная модель процесса включает:

- код программы;
- данные и глобальные переменные;
- кучу;
- стек(и) потоков;
- таблицу открытых файлов;
- идентификаторы процесса и родителя;
- права доступа и учетные данные;
- состояние планирования: готов, выполняется, ожидает, остановлен, завершен/zombie.

В терминах ядра Linux процесс представлен дескриптором процесса `struct task_struct`. Эта структура содержит почти всю информацию, которая нужна ядру для управления задачей:

- состояние процесса;
- идентификаторы PID и связи с родителем/потомками;
- приоритет и параметры планирования;
- указатели на адресное пространство;
- сведения об открытых файлах;
- обработчики и ожидающие сигналы;
- учетную информацию и права;
- ссылки, связывающие процесс со списками задач ядра.

В Linux поток не является принципиально другой сущностью ядра: и процессы, и потоки представлены задачами. Различие определяется тем, какие ресурсы разделяются между задачами. Например, потоки одного процесса разделяют адресное пространство и многие ресурсы, но имеют разные стеки и контекст выполнения.

### Создание процесса

На Unix-подобных системах новый процесс обычно создается через `fork()`.

- `fork()` создает копию процесса-родителя.
- После `fork()` у родителя и потомка разные PID.
- Современные Unix-системы обычно используют copy-on-write: страницы памяти реально копируются только при записи.
- Часто сразу после `fork()` потомок вызывает `execve()`, чтобы заменить образ процесса другой программой.

В Linux семейство вызовов создания задач опирается на механизм `clone()`: он позволяет точно указать, какие ресурсы новая задача будет разделять с родителем. На этом строится различие между классическим процессом после `fork()` и потоком, который разделяет адресное пространство, файловые дескрипторы и другие ресурсы.

### Время жизни процесса

Процесс существует от момента создания до завершения.

- Пока процесс работает, он может создавать дочерние процессы.
- Если родитель завершился раньше потомка, потомок становится сиротой и обычно переходит под управление `init`/`systemd`.

### Завершение процесса

Процесс завершает работу:

- явно через `exit()` / `return` из `main()`;
- по сигналу;
- из-за аварийной ошибки.

После завершения процесс может временно остаться в состоянии zombie: код уже не выполняется, но запись в таблице процессов хранит статус завершения. Родитель должен вызвать `wait()` / `waitpid()` и забрать этот статус.

С точки зрения ядра завершение означает освобождение ресурсов процесса и перевод задачи в состояние, из которого родитель может получить код завершения. Если родитель не забирает статус, остается zombie-запись: она уже не занимает пользовательское адресное пространство, но нужна для передачи результата завершения.

## 2. Межпроцессное взаимодействие. Состязательные ситуации. Критические области

**Межпроцессное взаимодействие (IPC)** — обмен данными и синхронизация между процессами.

Основные механизмы IPC:

- очереди сообщений System V;
- семафоры System V;
- разделяемая память System V;
- каналы, FIFO, сокеты, сигналы, файлы и `mmap()`.

### Состязательные ситуации

Состязательная ситуация возникает, когда несколько процессов или потоков одновременно обращаются к общему ресурсу, и результат зависит от порядка выполнения.

Это приводит к ошибкам:

- потеря обновлений;
- чтение несогласованных данных;
- нарушение инвариантов структуры данных.

### Критическая область

Критическая область — участок кода, который обращается к общему ресурсу и должен выполняться взаимно исключающим образом.

Требования к решению:

- взаимное исключение;
- отсутствие тупиков;
- отсутствие голодания;
- ограниченное ожидание.

Критическую область обычно защищают семафорами, мьютексами, спин-блокировками или другими примитивами синхронизации.

Синхронизация ядра нужна для защиты разделяемых данных ядра от одновременного доступа. Для коротких критических секций в ядре часто используются спин-блокировки, особенно когда нельзя спать. Если поток выполнения может блокироваться и ждать, применяются семафоры или мьютексы. Для пользовательских процессов в этих вопросах основное внимание уделяется System V IPC, но общая проблема та же: нужно защитить общий ресурс от гонок.

## 3. Взаимодействие процессов с помощью очередей сообщений. Очереди сообщений System V

Очередь сообщений System V — механизм IPC, где сообщения хранятся в ядре, а процессы отправляют и принимают их через идентификатор очереди.

### Назначение

- асинхронная передача данных;
- буферизация сообщений в ядре;
- разделение отправителя и получателя по времени.

### Создание и получение доступа

```c
int msgget(key_t key, int msgflg);
```

- `key` — ключ очереди, то есть имя IPC-объекта в пространстве System V;
- `msgflg` — флаги доступа и создания.

Основные флаги:

- `IPC_CREAT` — создать очередь, если ее нет;
- `IPC_EXCL` — создать только новую очередь, иначе ошибка;
- права доступа в младших 9 битах.

### Отправка сообщения

```c
int msgsnd(int msqid, const void *msgp, size_t msgsz, int msgflg);
```

Сообщение обычно имеет вид:

```c
struct msgbuf {
	long mtype;
	char mtext[...];
};
```

- `mtype` — положительный тип сообщения;
- `mtext` — данные сообщения;
- `msgsz` в `msgsnd()` и `msgrcv()` задает размер поля `mtext`, а не всей структуры вместе с `mtype`.

### Прием сообщения

```c
ssize_t msgrcv(int msqid, void *msgp, size_t msgsz, long msgtyp, int msgflg);
```

Смысл `msgtyp`:

- `0` — первое сообщение в очереди;
- `> 0` — первое сообщение указанного типа;
- `< 0` — сообщение с наименьшим типом, не превосходящим `|msgtyp|`; если таких несколько, берется самое старое среди сообщений этого типа.

Полезные флаги:

- `IPC_NOWAIT` — не блокироваться, если операция сейчас невозможна;
- `MSG_NOERROR` — при приеме обрезать слишком длинное сообщение вместо ошибки.

### Управление очередью

```c
int msgctl(int msqid, int cmd, struct msqid_ds *buf);
```

Команды:

- `IPC_STAT` — получить атрибуты очереди;
- `IPC_SET` — изменить атрибуты;
- `IPC_RMID` — удалить очередь.

### Свойства

- очередь хранится в ядре;
- процессы не обязаны одновременно быть активны;
- сообщения имеют тип и данные;
- можно реализовать приоритет по типам;
- очередь существует, пока ее явно не удалят через `msgctl(..., IPC_RMID, ...)` или пока система не будет перезагружена.

### Примеры из мануалов

Официальные справочные страницы подтверждают: `msgget()` создает или открывает очередь, `msgsnd()` отправляет, `msgrcv()` получает, а `msgctl()` управляет очередью.

## 4. Синхронизация процессов с помощью семафоров System V

**Семафор** — счетчик, используемый для синхронизации доступа к ресурсу.

### Назначение

- взаимное исключение;
- ожидание события;
- координация процессов.

### Основные свойства

- значение семафора — целое число;
- операции над семафором атомарны;
- `semop()` может блокировать процесс, пока условие не станет выполнимым.

### Создание и получение доступа

```c
int semget(key_t key, int nsems, int semflg);
```

Флаги:

- `IPC_CREAT` — создать набор, если его нет;
- `IPC_EXCL` — ошибка, если набор уже существует;
- права доступа в младших 9 битах.

### Операции над семафорами

```c
int semop(int semid, struct sembuf *sops, size_t nsops);
```

Структура операции:

```c
struct sembuf {
	unsigned short sem_num;
	short sem_op;
	short sem_flg;
};
```

- `sem_op < 0` — уменьшить семафор на `|sem_op|`; если текущее значение меньше нужного и нет `IPC_NOWAIT`, процесс ждет;
- `sem_op = 0` — ждать, пока значение станет нулем;
- `sem_op > 0` — увеличить значение семафора.

Флаги:

- `IPC_NOWAIT` — не блокироваться, если операция не может быть выполнена сразу;
- `SEM_UNDO` — ядро запоминает изменение и автоматически компенсирует его при завершении процесса.

### Управление набором семафоров

```c
int semctl(int semid, int semnum, int cmd, ...);
```

Основные команды:

- `IPC_STAT` — получить информацию;
- `IPC_SET` — изменить управляющие параметры;
- `IPC_RMID` — удалить набор;
- `GETVAL` — получить значение семафора;
- `SETVAL` — установить значение;
- `GETALL` / `SETALL` — получить/установить все значения;
- `GETNCNT` — число процессов, ожидающих увеличения значения;
- `GETZCNT` — число процессов, ожидающих нулевого значения.

### Поведение

- набор семафоров создается в ядре;
- значение семафора задает состояние синхронизации;
- для инициализации часто используют `semctl(..., SETVAL/SETALL, ...)`.

### Пример из лекции

Для сервер-клиентной синхронизации в лекции используется набор из двух семафоров. Начальное состояние `0 | 1`: первый семафор сигнализирует о наличии запроса для сервера, второй ограничивает доступ клиентов к разделяемому ресурсу.

## 5. Взаимодействие процессов с помощью разделяемой памяти System V

**Разделяемая память** — область памяти, доступная нескольким процессам одновременно.

Это самый быстрый механизм IPC для обмена большими объемами данных, потому что копирование через ядро минимально.

### Создание

```c
int shmget(key_t key, size_t size, int shmflg);
```

- `IPC_CREAT` — создать сегмент;
- `IPC_EXCL` — ошибка, если сегмент уже существует;
- права доступа задаются в младших 9 битах `shmflg`.

### Присоединение

```c
void *shmat(int shmid, const void *shmaddr, int shmflg);
```

- возвращает адрес, по которому сегмент отображен в адресное пространство процесса;
- при ошибке возвращает `(void *) -1`;
- флаг `SHM_RDONLY` присоединяет сегмент только для чтения.

### Отсоединение

```c
int shmdt(const void *shmaddr);
```

- отсоединяет сегмент от процесса.

### Управление

```c
int shmctl(int shmid, int cmd, struct shmid_ds *buf);
```

Команды:

- `IPC_STAT` — получить атрибуты;
- `IPC_SET` — изменить атрибуты;
- `IPC_RMID` — пометить сегмент на удаление.

### Свойства

- несколько процессов видят один и тот же участок данных;
- для согласованного доступа нужна синхронизация, чаще всего семафорами;
- `shmdt()` только отсоединяет сегмент от конкретного процесса;
- `IPC_RMID` помечает сегмент на удаление, а фактически память освобождается после последнего отсоединения.

## 6. Алгоритм синхронизации процессов на семафорах и разделяемой памяти

Типовая схема из лекции: один серверный процесс и несколько клиентских. Используются разделяемый ресурс и набор из двух семафоров с начальными значениями `0 | 1`.

Идея:

- разделяемая память используется как буфер данных;
- семафоры используются как сигналы состояния.

### Схема работы

1. Сервер создает разделяемую память и набор из двух семафоров, затем устанавливает значения `0 | 1`.
2. Сервер ждет данных от клиента операцией над первым семафором: условно `semop(-1, 0)`.
3. Клиент захватывает доступ к разделяемой памяти через второй семафор: `semop(0, -1)`, состояние становится `0 | 0`.
4. Клиент записывает запрос и сигнализирует серверу: `semop(1, 0)`, состояние становится `1 | 0`.
5. Клиент ждет ответа сервера операцией `semop(-1, -1)`.
6. Сервер читает запрос, обрабатывает его, записывает ответ и выполняет `semop(1, 1)`, состояние становится `1 | 1`.
7. Сервер снова переходит к ожиданию нового запроса.
8. Клиент разблокируется, читает ответ и освобождает доступ к ресурсу: `semop(0, 1)`, состояние возвращается к `0 | 1`.

### Зачем нужны оба механизма

- разделяемая память дает быстрый обмен данными;
- семафоры обеспечивают правильный порядок доступа.

Без семафоров возможны гонки: один процесс прочитает данные до того, как другой их полностью запишет.

## 7. Файлы и файловая система. Виртуальная файловая система (VFS): определение и назначение. Описание объектов VFS

**Файл** — именованная область данных и метаданных в файловой системе.

**Файловая система** — способ хранения, организации и доступа к файлам.

**VFS** (Virtual File System) — абстрактный слой ядра, который предоставляет единый интерфейс для разных реальных файловых систем.

VFS можно рассматривать как объектную модель, реализованную на C: ядро работает с набором общих объектов и таблиц операций, а конкретная файловая система подставляет свои функции. Благодаря этому системные вызовы вроде `open()`, `read()` и `write()` не зависят напрямую от ext4, tmpfs, procfs или другой конкретной файловой системы.

### Назначение VFS

- скрыть различия между ext4, xfs, tmpfs и другими ФС;
- дать единый интерфейс для `open`, `read`, `write`, `close` и т.д.;
- связать операции с объектами файловой системы.

### Основные объекты VFS

- **Суперблок** — информация о смонтированной файловой системе в целом;
- **inode** (файловый индекс, индексный дескриптор) — метаданные объекта файловой системы без имени;
- **dentry** — элемент каталога, связывающий имя с inode и ускоряющий разрешение путей;
- **file** — объект открытого файла: режим открытия, текущая позиция, ссылки на операции и inode/dentry.

## 8. Операции над объектами VFS

Каждый объект VFS имеет набор операций.

### Для суперблока

- создание и уничтожение inode;
- запись информации о ФС;
- синхронизация состояния;
- размонтирование и освобождение ресурсов файловой системы.

### Для inode

- поиск элемента;
- создание;
- удаление;
- переименование;
- получение и изменение атрибутов.

### Для dentry

- разрешение имени;
- сравнение имен;
- управление кэшированием каталогов.

### Для file

- `read`;
- `write`;
- `open`;
- `release` / `close`;
- `llseek`;
- `mmap`;
- `ioctl` при наличии специфичных операций устройства или файла.

### Связь с лекцией

В VFS объект хранит указатель на структуру операций, а ядро вызывает нужную функцию через этот указатель.

`super_operations`, `inode_operations`, `dentry_operations` и `file_operations` играют роль таблиц виртуальных методов. Например, вызов `read()` на уровне пользователя приводит к проходу через VFS и далее к конкретной функции чтения, зарегистрированной файловой системой или драйвером.

## 9. Файлы и их атрибуты. Операции над файлами

### Атрибуты файла

- тип файла;
- права доступа;
- количество жестких ссылок;
- UID владельца;
- GID владельца;
- размер;
- времена доступа, модификации и изменения атрибутов;
- номер inode;
- идентификатор файловой системы.

### Операции над файлами

- `open()` — открыть файл;
- `close()` — закрыть;
- `read()` — читать;
- `write()` — писать;
- `lseek()` — изменить текущую позицию;
- `stat()` / `fstat()` — получить атрибуты;
- `unlink()` — удалить имя файла;
- `rename()` — переименовать;
- `chmod()`, `chown()` — изменить права и владельца.

### Что происходит при `open()` и `close()`

Из лекции:

- `open()` ищет свободный дескриптор в таблице процесса и свободную запись в таблице открытых файлов ядра;
- запись открытого файла хранит режим открытия, текущую позицию чтения/записи и ссылку на inode/dentry;
- `close()` освобождает дескриптор процесса и уменьшает счетчики ссылок;
- `unlink()` удаляет имя из каталога, но данные файла обычно освобождаются только когда нет ни жестких ссылок, ни открытых файловых описаний.

## 10. Файлы, проецируемые в память. Обмен данными между процессами с помощью файлов, проецируемых в память

**Отображение файла в память** — это создание соответствия между участком виртуальной памяти и содержимым файла.

### Основные вызовы

```c
void *mmap(void *addr, size_t length, int prot, int flags, int fd, off_t offset);
int munmap(void *addr, size_t length);
```

### Зачем это нужно

- быстрый доступ к файлам через обычные операции с памятью;
- удобный обмен данными между процессами через один и тот же отображенный файл;
- меньше явных вызовов `read`/`write`.

### Типы отображения

- `MAP_SHARED` — изменения видны другим процессам и могут попасть в файл;
- `MAP_PRIVATE` — copy-on-write, изменения не идут в исходный файл.

### Обмен данными между процессами

Если два процесса отображают один и тот же файл с `MAP_SHARED`, то они могут обмениваться данными через общую область памяти.

Для согласованности обычно дополнительно используют:

- семафоры;
- барьеры;
- мьютексы, размещенные в разделяемой памяти;
- `msync()` при необходимости принудительно синхронизировать изменения с файлом на диске.

Важно различать видимость изменений между процессами и запись на диск: `MAP_SHARED` делает изменения общими для отображений, а `msync()` нужен именно для управления сбросом измененных страниц в файл.

### Источник

Справка `mmap(2)` подтверждает, что отображение создается в виртуальном адресном пространстве процесса, а `munmap()` удаляет отображение.

## 11. Алгоритмы управления динамической памятью

Динамический распределитель памяти обслуживает запросы вида "выделить блок размера `n`" и "освободить ранее выделенный блок". Он работает с областью памяти, разбитой на блоки. Каждый блок обычно содержит полезную часть и служебную информацию: размер, признак занятости, ссылки на соседние свободные блоки или другие поля, нужные конкретному алгоритму.

Главные задачи распределителя:

- быстро находить подходящий свободный блок;
- уменьшать внешнюю фрагментацию, когда свободная память есть, но разбита на мелкие непригодные куски;
- уменьшать внутреннюю фрагментацию, когда пользователю выдан блок больше запрошенного;
- быстро освобождать память и объединять соседние свободные блоки;
- хранить служебные данные так, чтобы они не занимали слишком много памяти.

Типовая операция выделения:

1. Запрошенный размер округляется до выравнивания, например до размера машинного слова.
2. Распределитель ищет свободный блок достаточного размера.
3. Если блок значительно больше запроса, он делится на занятый блок и свободный остаток.
4. Занятый блок помечается как используемый и возвращается вызывающему коду.

Типовая операция освобождения:

1. По адресу пользовательской области находится заголовок блока.
2. Блок помечается свободным.
3. Если соседние по адресу блоки тоже свободны, они объединяются.
4. Получившийся свободный блок помещается в структуру свободных блоков.

В контексте ядра Linux важно различать два уровня управления памятью:

- **физические страницы** управляются страничным распределителем, основанным на buddy system;
- **малые объекты ядра** часто выделяются через slab-подобные распределители, которые переиспользуют заранее подготовленные объекты и уменьшают стоимость частых выделений.

Экзаменационные методы first fit, best fit, worst fit и boundary tags описывают общие алгоритмы динамического распределения. Buddy system и деревья свободных блоков ближе к практическим системным распределителям.

### Метод первого подходящего

**First fit** выбирает первый свободный блок, размер которого не меньше запрошенного.

Алгоритм выделения:

1. Свободные блоки просматриваются с начала списка.
2. Первый блок размера `>= n` выбирается для выделения.
3. Если остаток после выделения достаточно велик, блок делится.
4. Остаток остается в списке свободных блоков, а выделенная часть помечается занятой.

Плюсы:

- простая реализация;
- часто работает быстрее полного просмотра списка;
- при удачном распределении запросов дает приемлемую фрагментацию.

Минусы:

- начало списка постепенно заполняется мелкими остатками;
- качество выбора зависит от порядка блоков в списке;
- в худшем случае поиск остается линейным: `O(m)`, где `m` — число свободных блоков.

### Метод лучшего подходящего

**Best fit** выбирает самый маленький свободный блок, который все еще достаточен для запроса.

Алгоритм выделения:

1. Просматриваются все свободные блоки или используется структура поиска по размеру.
2. Среди блоков размера `>= n` выбирается блок с минимальным размером.
3. Если после выделения остается достаточно большой кусок, он возвращается в список свободных блоков.

Плюсы:

- уменьшает остаток после конкретного выделения;
- хорошо использует память при некоторых наборах запросов;
- удобно реализуется через дерево свободных блоков, упорядоченное по размеру.

Минусы:

- при линейном списке требует полного просмотра: `O(m)`;
- часто порождает много очень маленьких свободных фрагментов;
- локально лучший выбор не всегда минимизирует фрагментацию в будущем.

### Метод худшего подходящего

**Worst fit** выбирает самый большой свободный блок.

Алгоритм выделения:

1. Находится максимальный свободный блок.
2. Из него выделяется нужный размер.
3. Остаток, если он достаточно велик, остается свободным.

Идея метода: если резать самый большой блок, то остаток тоже с большей вероятностью будет пригоден для следующих запросов.

Плюсы:

- стремится не создавать совсем маленькие остатки;
- может работать неплохо, если запросы имеют близкие размеры.

Минусы:

- быстро разрушает крупные свободные области;
- при линейном поиске требует полного просмотра списка;
- на практике часто хуже first fit и best fit.

### Освобождение в рассортированном списке

В этом методе свободные блоки хранятся в списке, обычно упорядоченном по адресу начала блока. Основная цель сортировки по адресу — быстро понять, какие свободные блоки являются физическими соседями в памяти.

Алгоритм освобождения:

1. Освобождаемый блок помечается свободным.
2. В отсортированном списке находится место, куда его нужно вставить по адресу.
3. Проверяется предыдущий свободный блок: если его конец совпадает с началом освобождаемого блока, блоки объединяются.
4. Проверяется следующий свободный блок: если конец освобождаемого блока совпадает с началом следующего, они тоже объединяются.
5. В списке остается один укрупненный свободный блок.

Плюсы:

- слияние соседних свободных блоков уменьшает внешнюю фрагментацию;
- структура проста;
- не нужны сложные балансирующие деревья.

Минусы:

- поиск места вставки в односвязном или двусвязном списке занимает `O(m)`;
- поиск блока для выделения тоже обычно линейный;
- при большом числе свободных блоков операции становятся дорогими.

### Метод освобождения с дескрипторами границ

Метод с дескрипторами границ, или **boundary tags**, хранит служебные поля в начале и в конце каждого блока. Заголовок и подвал обычно содержат размер блока и признак занятости. Благодаря этому при освобождении можно быстро найти соседние блоки по адресу, не проходя весь список.

Типичная структура блока:

```text
[ header: size, used/free ][ payload ... ][ footer: size, used/free ]
```

Алгоритм освобождения:

1. По адресу пользовательских данных находится заголовок текущего блока.
2. По граничному дескриптору слева определяется размер и состояние предыдущего физического блока.
3. По заголовку справа определяется размер и состояние следующего физического блока.
4. Если сосед слева свободен, текущий блок объединяется с ним.
5. Если сосед справа свободен, текущий блок объединяется с ним.
6. Итоговый блок вставляется в список свободных блоков.

Плюсы:

- объединение с соседями выполняется за `O(1)`, если есть прямой доступ к соседним дескрипторам;
- удобно хранить размер блока и флаги занятости;
- проще контролировать целостность структуры памяти.

Минусы:

- служебные поля увеличивают накладные расходы;
- повреждение заголовка или подвала может разрушить структуру распределителя;
- нужно аккуратно поддерживать согласованность заголовка и подвала при каждом делении и слиянии.

### Система двойников (buddy system)

**Система двойников** делит память на блоки размеров, равных степеням двойки. Для каждого допустимого размера обычно хранится отдельный список свободных блоков.

Алгоритм выделения:

1. Запрошенный размер округляется вверх до ближайшей допустимой степени двойки с учетом служебных данных.
2. В списке этого размера ищется свободный блок.
3. Если такого блока нет, ищется ближайший больший свободный блок.
4. Больший блок делится пополам на двух двойников.
5. Деление повторяется, пока не получится блок нужного размера.
6. Один блок выдается пользователю, остальные половины помещаются в списки свободных блоков соответствующих размеров.

Алгоритм освобождения:

1. Блок помечается свободным.
2. Вычисляется адрес его двойника. Для блока размера `2^k` двойник отличается от него битом `k` в адресе относительно начала управляемой области.
3. Если двойник свободен и имеет тот же размер, два блока объединяются в блок размера `2^(k+1)`.
4. Слияние повторяется вверх по размерам, пока двойник занят или пока не достигнут максимальный размер.
5. Итоговый блок помещается в список свободных блоков своего размера.

Плюсы:

- быстрый поиск подходящего класса размера;
- быстрое слияние, потому что адрес двойника вычисляется напрямую;
- простая структура списков по степеням двойки;
- хорошо подходит для системных распределителей страниц и низкоуровневой памяти.

В Linux buddy allocator используется именно для управления физическими страницами. Память разбивается на блоки порядка `order`, где размер блока равен `2^order` страницам. При нехватке блока нужного порядка ядро берет блок большего порядка и делит его на двойников.

Минусы:

- внутренняя фрагментация: запрос на `513` байт может занять блок `1024` байта;
- размеры блоков жестко ограничены степенями двойки;
- для мелких объектов часто нужны дополнительные распределители поверх buddy system.

Поверх buddy allocator в ядре применяются объектные распределители, например slab/slub-подход: они создают кэши объектов одного типа или размера. Это уменьшает фрагментацию и ускоряет повторное выделение часто используемых структур ядра, таких как дескрипторы процессов, inode, dentry и другие объекты.

### Двусвязный список всех блоков и красно-черное дерево свободных блоков

Этот метод сочетает две структуры:

- **двусвязный список всех блоков** в порядке расположения в памяти;
- **красно-черное дерево свободных блоков**, упорядоченное по размеру или по паре `(размер, адрес)`.

Зачем нужен список всех блоков:

- при освобождении можно быстро найти физически предыдущий и следующий блок;
- удобно выполнять слияние соседних свободных блоков;
- список отражает реальный порядок блоков в памяти.

Зачем нужно красно-черное дерево:

- свободный блок можно искать за `O(log m)`;
- удобно реализовать best fit: найти минимальный свободный блок размера `>= n`;
- вставка и удаление свободного блока тоже выполняются за `O(log m)`.

Алгоритм выделения:

1. В красно-черном дереве находится подходящий свободный блок.
2. Этот блок удаляется из дерева свободных блоков.
3. Если блок слишком большой, он делится: одна часть выдается пользователю, остаток становится новым свободным блоком.
4. В двусвязном списке всех блоков обновляются связи между соседями.
5. Свободный остаток вставляется обратно в красно-черное дерево.

Алгоритм освобождения:

1. Блок помечается свободным.
2. Через двусвязный список проверяются левый и правый физические соседи.
3. Свободные соседи удаляются из красно-черного дерева.
4. Блоки объединяются в один большой свободный блок.
5. Объединенный блок вставляется в красно-черное дерево.

Итог: список дает быстрое слияние по адресу, дерево дает быстрый поиск по размеру. Это сложнее простого списка, но масштабируется лучше при большом числе блоков.

## 12. Сбалансированные по высоте деревья поиска. Повороты. B-дерево, B+-дерево, B*-дерево, B*+-дерево

## 1. Сбалансированные по высоте деревья поиска (АВЛ-деревья)

**АВЛ-дерево** — это двоичное дерево поиска, для которого выполняется свойство сбалансированности по высоте: для любой вершины дерева высота её правого поддерева отличается от высоты левого поддерева не более чем на 1.
Показатель баланса (Balance Factor, BF) = Высота(Левого поддерева) - Высота(Правого поддерева). Допустимые значения: -1, 0, 1.

### Полный спектр операций над АВЛ-деревом:
1. **Поиск (Search) O(log n)**: Стандартный бинарный поиск.
2. **Вставка (Insert) O(log n)**: Стандартная вставка, после которой происходит возврат вверх к корню с пересчетом баланса (BF) и выполнением поворотов при $|BF| > 1$.
3. **Удаление (Delete) O(log n)**: Удаление узла с последующим подъемом вверх к корню с балансировкой (может потребовать до $O(\log n)$ поворотов).
4. **Поиск минимума/максимума (Find Min/Max) O(log n)**: Спуск максимально влево или вправо.
5. **Поиск следующего/предыдущего (Successor/Predecessor) O(log n)**.
6. **Обход (Traversal) O(n)**: In-order, Pre-order и Post-order.

### Архитектура узла и функции высоты
```cpp
#include <algorithm>

template <typename Key>
struct AVLNode {
    Key key;
    int height;
    AVLNode* left;
    AVLNode* right;
};

int height(AVLNode<Key>* node) { return node ? node->height : 0; }
void updateHeight(AVLNode<Key>* node) {
    if (node) node->height = 1 + std::max(height(node->left), height(node->right));
}
```

### Операции поворотов (для восстановления баланса)

*Примечание: "Большой левый" синонимичен "Двойному право-левому", а "Большой правый" — "Двойному лево-правому".*

```cpp
// 1. МАЛЫЙ ПРАВЫЙ ПОВОРОТ (Small Right Rotation / LL-случай)
// Выполняется, если левое поддерево выше правого на 2, и дисбаланс вызван левым потомком левого узла.
AVLNode<Key>* smallRightRotate(AVLNode<Key>* y) {
    AVLNode<Key>* x = y->left;
    y->left = x->right;
    x->right = y;
    
    updateHeight(y);
    updateHeight(x);
    return x; // x - новый корень
}

// 2. МАЛЫЙ ЛЕВЫЙ ПОВОРОТ (Small Left Rotation / RR-случай)
// Выполняется, если правое поддерево выше левого на 2, и дисбаланс вызван правым потомком правого узла.
AVLNode<Key>* smallLeftRotate(AVLNode<Key>* x) {
    AVLNode<Key>* y = x->right;
    x->right = y->left;
    y->left = x;
    
    updateHeight(x);
    updateHeight(y);
    return y; // y - новый корень
}

// 3. ДВОЙНОЙ / БОЛЬШОЙ ПРАВЫЙ ПОВОРОТ (LR-случай)
// Малый левый поворот для левого потомка, затем малый правый для текущего узла.
AVLNode<Key>* bigRightRotate(AVLNode<Key>* node) {
    node->left = smallLeftRotate(node->left);
    return smallRightRotate(node);
}

// 4. ДВОЙНОЙ / БОЛЬШОЙ ЛЕВЫЙ ПОВОРОТ (RL-случай)
// Малый правый поворот для правого потомка, затем малый левый для текущего узла.
AVLNode<Key>* bigLeftRotate(AVLNode<Key>* node) {
    node->right = smallRightRotate(node->right);
    return smallLeftRotate(node);
}
```

## 2. Архитектура семейства B-деревьев

B-дерево и его вариации — это сильно ветвящиеся сбалансированные деревья поиска, разработанные для оптимизации хранения данных на диске и во внешней памяти. В отличие от бинарных деревьев (АВЛ, красно-черных), где каждый узел содержит один ключ и два потомка, B-дерево позволяет хранить множество ключей в одном узле и иметь множество потомков.

### Почему B-деревья эффективны для дисков

Когда данные хранятся на диске, доступ к памяти происходит блоками (страницами). Один узел B-дерева обычно соответствует одной дисковой странице (4KB, 8KB и т.д.). Это означает, что одна операция чтения с диска загружает сразу множество ключей, что минимизирует дорогие операции ввода-вывода.

### Структура и свойства B-дерева

B-дерево порядка $t$ (минимальная степень) обладает следующими свойствами:

1. **Все листья на одной глубине** — дерево идеально сбалансировано
2. **Каждый узел (кроме корня) содержит от $t-1$ до $2t-1$ ключей**
3. **Каждый внутренний узел с $k$ ключами имеет $k+1$ потомков**
4. **Корень содержит от 1 до $2t-1$ ключей** (может иметь меньше $t-1$)
5. **Ключи в узле отсортированы по возрастанию**

**Пример**: При $t=2$ (2-3-4 дерево) узел содержит 1-3 ключа и 2-4 потомка. При $t=100$ узел содержит 99-199 ключей и 100-200 потомков — высота дерева остается маленькой даже для миллионов записей.

### Балансировка в B-деревьях

В отличие от АВЛ-деревьев, где балансировка выполняется через **повороты**, в B-деревьях используются другие механизмы:

#### Почему нет классических поворотов?

В бинарных деревьях повороты работают с узлами, содержащими один ключ. В B-дереве узел содержит множество ключей, поэтому "поворот" одного ключа нарушил бы структуру многоключевого узла. Вместо этого используются операции, сохраняющие инварианты многоключевых узлов.

#### Три механизма поддержания баланса:

**1. Сплит (Split) — рост вверх**

Когда узел переполняется (содержит $2t-1$ ключей), он делится на два:
- Средний ключ поднимается в родительский узел
- Левая половина остается в текущем узле
- Правая половина создается как новый узел

Это единственная операция, которая **увеличивает высоту дерева** — когда корень переполняется и делится, создается новый корень.

**2. Мерж (Merge) — слияние при удалении**

Когда узел становится слишком маленьким (меньше $t-1$ ключей после удаления), он объединяется с соседним узлом через родительский ключ:
- Ключ-разделитель из родителя спускается вниз
- Два узла сливаются в один
- Если родитель становится слишком маленьким, операция повторяется рекурсивно вверх

**3. Перераспределение (Redistribute/Borrow) — переливание ключей**

Вместо слияния можно "перелить" ключ из соседнего узла через родителя:
- Если у соседа есть лишние ключи (больше $t-1$), один ключ перемещается через родителя
- Это предотвращает слишком частое слияние и расщепление
- Сохраняет локальность данных

### Отличия вариантов B-деревьев

**B-дерево (классическое)**:
- Данные хранятся во всех узлах (и внутренних, и листьях)
- Минимальное заполнение 1/2
- Поиск может закончиться в любом узле

**B+ дерево**:
- Данные хранятся **только в листьях**
- Внутренние узлы содержат только ключи-разделители для навигации
- Листья связаны в односвязный или двусвязный список
- **Преимущество**: быстрый диапазонный поиск — после нахождения первого элемента можно последовательно читать листья без возврата к корню

**B* дерево**:
- Минимальное заполнение **2/3** (вместо 1/2)
- При переполнении сначала пытается **перераспределить** ключи с соседом
- Сплит происходит только когда оба узла полны — тогда **два узла делятся на три** (2→3)
- **Преимущество**: более плотное заполнение, меньше расход памяти

**B*+ дерево**:
- Комбинация B+ и B*: данные только в листьях + заполнение 2/3 + перераспределение
- Наиболее эффективно для баз данных и файловых систем
- Используется в NTFS, ReiserFS, многих СУБД

### Поиск в B-дереве

Алгоритм поиска ключа $k$:

1. Начинаем с корня
2. В текущем узле ищем ключ $k$ среди отсортированных ключей
3. Если нашли — возвращаем результат
4. Если не нашли и это лист — ключа нет в дереве
5. Если не нашли и это внутренний узел — выбираем потомка между ключами, между которыми находится $k$
6. Повторяем шаги 2-5

**Сложность**: $O(\log_t n)$ — логарифм по основанию $t$, что очень эффективно при больших $t$.

### Вставка в B-дерево

1. Находим лист, куда должен попасть новый ключ
2. Если в листе есть место (меньше $2t-1$ ключей) — просто вставляем
3. Если лист полон:
   - Делим лист на два
   - Средний ключ поднимаем в родителя
   - Если родитель тоже полон — делим рекурсивно вверх
   - Если корень полон — дерево растет в высоту

### Вставка в B*+ дерево (с перераспределением)

B* дерево использует более сложную стратегию:

1. Находим лист для вставки
2. Если есть место — вставляем
3. Если лист полон:
   - **Проверяем соседа**: если у соседа есть место, перераспределяем ключи между соседями через родителя
   - Если оба соседа полны (или один из двух полон при нечетном числе соседей) — выполняем **сплит 2→3**: два полных узла и новый ключ делятся на три узла

**Почему перераспределение лучше сплита?**
- Не увеличивает число узлов
- Сохраняет высоту дерева
- Обеспечивает более равномерное заполнение

### Удаление из B-дерева

1. Если ключ в листе и после удаления узел остается достаточно заполненным (≥ $t-1$ ключей) — просто удаляем
2. Если ключ во внутреннем узле:
   - Заменяем на предшественника (максимум левого поддерева) или преемника (минимум правого поддерева)
   - Удаляем ключ из листа
3. Если после удаления узел недозаполнен (< $t-1$ ключей):
   - **Заимствование**: если у соседа есть лишние ключи, переносим один через родителя
   - **Слияние**: если заимствовать нельзя, сливаем с соседом, ключ родителя спускается в объединенный узел

### Почему B-деревья не используют повороты?

Повороты (rotations) — это операции бинарных деревьев. В B-деревьях:
- Узлы содержат **множество ключей**, не один
- Структура направлена на **минимизацию дисковых операций**, не на локальную балансировку
- Операции сплита/мержа/перераспределения работают с **целыми страницами** данных

Вместо "поворота" одного ключа, B-дерево перераспределяет ключи **между целыми узлами**, сохраняя инварианты заполнения ($t-1$ до $2t-1$ ключей).

### Сравнение с АВЛ-деревьями

| Характеристика | АВЛ-дерево | B-дерево |
|----------------|------------|----------|
| Ключей в узле | 1 | $t-1$ до $2t-1$ |
| Потомков | 2 | $k+1$ (где $k$ — число ключей) |
| Балансировка | Повороты | Сплит/мерж/редистрибьют |
| Использование | Оперативная память | Диск, внешняя память |
| Высота | $O(\log_2 n)$ | $O(\log_t n)$ |
| Диапазонный поиск | Требует обхода | Эффективен (B+) |

### Базовые низкоуровневые операции:
*   **Сплит (Split)**: При переполнении узла он делится на два. Дерево растет вверх.
*   **Мердж (Merge)**: При удалении, если узлы заполнены меньше минимума, два соседних сливаются в один.
*   **Заимствование / Перераспределение (Borrow/Redistribute)**: Элемент "перетекает" от переполненного соседа к текущему через родителя.

### Спецификации разных B-деревьев:
1. **B-дерево**: Ключи и данные хранятся везде (и в корнях, и в листьях). Узел должен быть заполнен минимум на 1/2.
2. **B+ дерево**: Все данные хранятся **только в листьях**. Внутренние узлы — лишь индексы-маршрутизаторы. Листья связаны в **список** для сверхбыстрого поиска в диапазоне.
3. **B* дерево**: Версия B-дерева, где узлы заполнены минимум на **2/3**. При переполнении сначала делается **redistribute** с братом, и только если оба полны — сплит (2 узла бьются на 3).
4. **B*+ дерево**: Объединяет B+ (данные только в листьях, листья в списке) и B* (заполнение 2/3, перераспределение вместо немедленного сплита).

### Кодовые операции B-деревьев

```cpp
BNode<Key>* searchKey(BNode<Key>* node, Key target) {
    if (!node) return nullptr;
    int i = 0;
    while (i < node->keys.size() && target > node->keys[i]) i++;
    if (i < node->keys.size() && node->keys[i] == target) return node;

}
```

**2. Поиск в диапазоне ключей (B+ и B*+)**
```cpp
std::vector<Key> rangeSearch(BNode<Key>* root, Key start, Key end) {
    std::vector<Key> result;
    if (!root) return result;

    // Спуск к листу за O(log N)
    BNode<Key>* curr = root;
    while (!curr->is_leaf) {
        int i = 0;
        while (i < curr->keys.size() && start > curr->keys[i]) i++;
        curr = curr->children[i];
    }

    // Проход по связному списку листьев
    while (curr != nullptr) {
        for (int i = 0; i < curr->keys.size(); i++) {
            if (curr->keys[i] >= start && curr->keys[i] <= end) result.push_back(curr->keys[i]);
            if (curr->keys[i] > end) return result; // Оптимизация выхода
        }
        curr = curr->next_leaf;
    }
    return result;
}
```

**3. Операции добавления (Сплит и Перераспределение)**
```cpp
// B и B+ дерево: Сплит узла
void splitChild(BNode<Key>* parent, int child_idx) {
    BNode<Key>* old_node = parent->children[child_idx];
    BNode<Key>* new_node = new BNode<Key>{old_node->is_leaf};
    
    if (old_node->is_leaf) { // Поддержка списка B+ дерева
        new_node->next_leaf = old_node->next_leaf;
        old_node->next_leaf = new_node;
    }
    // ... логика разделения пополам (t элементов) ...
    int t = old_node->keys.size() / 2;
    parent->keys.insert(parent->keys.begin() + child_idx, old_node->keys[t]);
    parent->children.insert(parent->children.begin() + child_idx + 1, new_node);
}

// B* и B*+ дерево: Перераспределение (Redistribute) 
bool redistribute(BNode<Key>* parent, int child_idx) {
    BNode<Key>* node = parent->children[child_idx];
    if (child_idx < parent->children.size() - 1) {
        BNode<Key>* right_sibling = parent->children[child_idx + 1];
        if (!right_sibling_is_full) { // Условное поле
            right_sibling->keys.insert(right_sibling->keys.begin(), node->keys.back());
            node->keys.pop_back();
            parent->keys[child_idx] = right_sibling->keys.front(); // Обновление родителя
            return true;
        }
    }
    return false; // Требуется Split 2 в 3
}

// B*+ дерево: Полная вставка с перераспределением и сплитом 2->3
class BStarPlusTree {
    static const int T = 2; // Минимальная степень, max_keys = 2*T
    BNode<int>* root;
    
    bool isFull(BNode<int>* node) {
        return node->keys.size() >= 2 * T;
    }
    
    void insertNonFull(BNode<int>* node, int key) {
        if (node->is_leaf) {
            // Вставка в лист
            auto it = std::lower_bound(node->keys.begin(), node->keys.end(), key);
            node->keys.insert(it, key);
        } else {
            // Поиск ребенка для спуска
            int i = node->keys.size() - 1;
            while (i >= 0 && key < node->keys[i]) i--;
            i++;
            
            BNode<int>* child = node->children[i];
            if (isFull(child)) {
                // Пытаемся перераспределить (B* стратегия)
                if (!tryRedistribute(node, i)) {
                    // Перераспределение невозможно - сплит 2->3
                    split2to3(node, i);
                }
                // Определяем куда идти после сплита/перераспределения
                if (key > node->keys[i]) i++;
            }
            insertNonFull(node->children[i], key);
        }
    }
    
    bool tryRedistribute(BNode<int>* parent, int idx) {
        BNode<int>* node = parent->children[idx];
        // Пробуем правого брата
        if (idx < parent->children.size() - 1) {
            BNode<int>* right = parent->children[idx + 1];
            if (!isFull(right)) {
                // Перемещаем ключ из node в right через parent
                right->keys.insert(right->keys.begin(), parent->keys[idx]);
                if (!node->is_leaf) {
                    right->children.insert(right->children.begin(), node->children.back());
                    node->children.pop_back();
                }
                parent->keys[idx] = node->keys.back();
                node->keys.pop_back();
                return true;
            }
        }
        // Пробуем левого брата
        if (idx > 0) {
            BNode<int>* left = parent->children[idx - 1];
            if (!isFull(left)) {
                left->keys.push_back(parent->keys[idx - 1]);
                if (!node->is_leaf) {
                    left->children.push_back(node->children[0]);
                    node->children.erase(node->children.begin());
                }
                parent->keys[idx - 1] = node->keys[0];
                node->keys.erase(node->keys.begin());
                return true;
            }
        }
        return false;
    }
    
    void split2to3(BNode<int>* parent, int idx) {
        BNode<int>* left = parent->children[idx];
        BNode<int>* right = parent->children[idx + 1];
        
        // Создаем средний узел
        BNode<int>* middle = new BNode<int>{left->is_leaf};
        
        // Объединяем все ключи left + parent->keys[idx] + right
        std::vector<int> allKeys = left->keys;
        allKeys.push_back(parent->keys[idx]);
        allKeys.insert(allKeys.end(), right->keys.begin(), right->keys.end());
        
        // Разделяем на три части (2/3 заполнения)
        int n = allKeys.size();
        int third = n / 3;
        
        left->keys.assign(allKeys.begin(), allKeys.begin() + third);
        middle->keys.assign(allKeys.begin() + third + 1, allKeys.begin() + 2 * third + 1);
        parent->keys[idx] = allKeys[third];
        
        if (left->is_leaf) {
            // Для листьев B+ - все данные остаются в листьях
            middle->next_leaf = right->next_leaf;
            left->next_leaf = middle;
            right->keys.assign(allKeys.begin() + 2 * third + 1, allKeys.end());
        } else {
            right->keys.assign(allKeys.begin() + 2 * third + 1, allKeys.end());
            // Перераспределяем детей
            std::vector<BNode<int>*> allChildren = left->children;
            allChildren.insert(allChildren.end(), right->children.begin(), right->children.end());
            left->children.assign(allChildren.begin(), allChildren.begin() + third + 1);
            middle->children.assign(allChildren.begin() + third + 1, allChildren.begin() + 2 * third + 2);
            right->children.assign(allChildren.begin() + 2 * third + 2, allChildren.end());
        }
        
        // Вставляем middle между left и right
        parent->keys.insert(parent->keys.begin() + idx + 1, allKeys[2 * third]);
        parent->children.insert(parent->children.begin() + idx + 1, middle);
    }
    
public:
    BStarPlusTree() : root(new BNode<int>{true}) {}
    
    void insert(int key) {
        if (isFull(root)) {
            // Корень полон - создаем новый корень
            BNode<int>* newRoot = new BNode<int>{false};
            newRoot->children.push_back(root);
            splitChild(newRoot, 0);
            root = newRoot;
        }
        insertNonFull(root, key);
    }
};
```

**4. Операции удаления (Заимствование и Слияние)**
```cpp
// Заимствование (Borrow) у левого брата
void borrowFromPrev(BNode<Key>* parent, int child_idx) {
    BNode<Key>* node = parent->children[child_idx];
    BNode<Key>* left_sibling = parent->children[child_idx - 1];

    node->keys.insert(node->keys.begin(), parent->keys[child_idx - 1]);
    parent->keys[child_idx - 1] = left_sibling->keys.back();
    left_sibling->keys.pop_back();
}

// Слияние (Merge) с правым братом
void mergeNodes(BNode<Key>* parent, int child_idx) {
    BNode<Key>* left = parent->children[child_idx];
    BNode<Key>* right = parent->children[child_idx + 1];

    left->keys.push_back(parent->keys[child_idx]); // Ключ из родителя
    for (auto k : right->keys) left->keys.push_back(k);
    
    if (left->is_leaf) left->next_leaf = right->next_leaf; // Поддержка B+

    parent->keys.erase(parent->keys.begin() + child_idx);
    parent->children.erase(parent->children.begin() + child_idx + 1);
    delete right;
}
```
